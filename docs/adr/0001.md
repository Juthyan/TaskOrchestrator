# ADR-0001 — ITaskRepository in Application layer

## Status
Accepted

## Context
The system needs to persist and retrieve tasks. The Domain layer contains
the business rules and must remain pure — no infrastructure dependencies.
The question was: where should the repository contract live?

## Decision
Place `ITaskRepository` in the Application layer, not in Domain or Infrastructure.
The Application layer defines the contract (interface), and Infrastructure
provides the concrete implementation (`InMemoryTaskRepository`).

## Reasons
- Dependency inversion — Application depends on an abstraction, not a concrete class
- Infrastructure can be swapped (InMemory → PostgreSQL) without touching Application or Domain
- Domain stays pure — no repository concepts leak into business rules
- The compiler enforces the boundary — Infrastructure must reference Application, not the other way around

## Consequences
- Every new storage implementation must implement `ITaskRepository`
- Application never imports from Infrastructure
- Testing is easier — any test can inject a fake repository without spinning up a database
